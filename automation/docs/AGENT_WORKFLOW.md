# 에이전트 작업 흐름

## 새 프로젝트를 연결할 때

1. 루트 `AGENTS.md`와 `SAFETY_RULES.md`를 읽는다.
2. `project:init`으로 프로젝트 폴더를 만든다.
3. FE/BE 소스와 실행 중인 서비스를 읽기 전용으로 살펴본다.
4. `project.json`에 URL, 로그인 선택자, 오류·로딩 기준을 채운다.
5. 바로 열어 볼 수 있는 화면은 `routes.json`에 정리한다.
6. 상세 화면에 동적 ID가 필요하면 프로젝트 `support/`에서 실제 API 응답으로 대상을 찾는다.
7. 검색, 필터, 탭, 페이지 이동, 다운로드는 `tests/`에 사용자 흐름으로 작성한다.
8. 아래 범위 점검표를 채우고, 해당 기능이 있는데 시나리오가 없으면 보강한다.
9. `validate`, `typecheck`, `audit` 순서로 실행한다.
10. 보고서를 읽고 제품 이슈와 환경 문제, 오탐을 구분한다.

## 범위 점검표

| 범위 | 확인 방법 | 완료 기준 |
| --- | --- | --- |
| 정적 경로 | FE 라우터와 메뉴를 `routes.json`과 대조 | 읽기 가능한 경로를 모두 포함하거나 제외 이유 기록 |
| 동적 상세 | `:id` 경로와 목록의 상세 링크 확인 | 실제 API 응답에서 ID를 골라 상세 데이터 표출 검증 |
| 검색 | 검색 입력과 요청 파라미터 확인 | 입력값이 화면 상태나 API 요청에 반영됨 |
| 필터·탭 | 날짜, 상태, 유형, 탭 확인 | 선택 상태와 URL 또는 API 요청이 함께 바뀜 |
| 페이지 이동 | 다음·이전 버튼과 페이지 파라미터 확인 | 이동과 복귀가 모두 동작함 |
| 다운로드 | 읽기 전용 다운로드 버튼 확인 | 파일명, 확장자, 파일 크기를 검증함 |
| API 계약 | 화면 구조를 결정하는 핵심 응답 확인 | 필수 ID, 이름, 목록 구조를 검증함 |
| 권한 | 사용자·관리자 경로 확인 | 테스트 계정으로 검사 가능한 범위와 제외 범위 기록 |

기능이 실제로 없거나 계정 권한 때문에 확인할 수 없다면 억지로 테스트를 만들지 않는다. 대신 최종 결과의 `실행하지 못한 범위`에 이유를 적는다.

## 기능이 바뀌었을 때

먼저 기존 시나리오가 바뀐 기능을 이미 다루는지 확인한다.

- 내부 구현만 바뀌었다면 기존 회귀검사를 실행한다.
- 사용자 행동이나 FE/API 계약이 추가됐다면 새 시나리오를 작성한다.
- 의도적으로 URL이나 응답 계약을 바꾼 경우에만 기존 기대값을 수정한다.
- 실패 원인이 제품에 있는데 테스트를 느슨하게 만들어 통과시키지 않는다.

## 결과를 전달할 때

다음 다섯 가지만 빠짐없이 적는다.

- 실행한 시나리오 수와 통과·실패 수
- 새로 발견한 이슈
- 이미 알려진 이슈
- 사람이 판단해야 할 항목
- 실행하지 못한 범위와 이유

상세 근거는 `reports/report.html`에서 확인할 수 있게 한다.
