# 새 프로젝트 연결하기

## 먼저 준비할 정보

- 영문 프로젝트 ID: 예) `billing-admin`
- FE와 BE 소스의 로컬 경로
- 실행 중인 FE와 API 주소
- 테스트 계정에 사용할 환경변수
- 로그인 성공 뒤 도착하는 경로
- E2E에서 확인할 핵심 사용자 흐름
- 로그인 과정에서 꼭 필요한 쓰기 요청

로그인 입력창은 화면에 보이는 글자만 보고 선택자를 정하지 않는다. 실제 접근성 이름을 확인하고 `role`을 우선 사용한다. `<label>` 연결이 확인된 경우에만 `label` 방식을 사용한다.

## 프로젝트 폴더 만들기

저장소 루트에서 실행한다.

```powershell
npm --prefix automation run project:init -- --project my-project
```

생성된 `automation/projects/my-project/`만 프로젝트에 맞게 수정한다. 공통 엔진 변경은 같은 요구가 다른 프로젝트에도 있는지 확인한 뒤 결정한다.

## 조사 순서

1. FE 라우터와 메뉴 구조
2. 로그인과 권한 처리
3. 공통 API 클라이언트와 환경변수
4. 목록, 상세, 검색, 필터, 페이지 이동
5. 다운로드
6. 오류 화면과 로딩 UI
7. 상세 화면에서 쓸 동적 ID를 찾는 방법

조사가 끝나면 다음 항목을 하나씩 `검사함` 또는 `제외함`으로 판단한다.

- FE 라우터의 모든 읽기 가능한 정적 경로
- 실제 데이터 ID가 필요한 상세 경로
- 검색
- 필터와 탭 전환
- 페이지 이동
- 파일 다운로드
- 화면 구조를 결정하는 핵심 API 계약
- 사용자·관리자 등 권한별 접근 범위

`핵심 흐름 한 개`를 만들었다는 이유로 조사를 끝내지 않는다. 기능이 있는데 시나리오가 없으면 보강하고, 실행할 수 없는 항목은 이유를 최종 결과에 적는다.

## 첫 실행 결과 정리

첫 실행에서 같은 오류가 여러 화면에 반복되면 먼저 한 가지 근본 원인인지 확인한다. 원인이 확인된 전역 오류는 `known-issues.json`에 짧고 안정적인 지문을 두 개 이상 등록한 뒤 다시 실행한다. 알려진 이슈 등록은 오류를 숨기는 기능이 아니라, 같은 원인을 보고서에서 한 건으로 묶고 나머지 시나리오를 끝까지 실행하기 위한 장치다.

로그인 선택자 오류, 서버 미실행, 잘못된 URL은 제품 이슈로 등록하지 않는다. 프로필 또는 환경 문제를 고친 뒤 전체 실행을 다시 시작한다.

화면만 보고 API 문제라고 단정하지 않는다. 화면에서 이상을 찾으면 네트워크 응답과 FE 호출부를 함께 확인하고, 필요할 때만 BE 구현까지 따라간다.

## 연결이 끝났다고 볼 수 있는 기준

- 프로필 검증 통과
- 테스트 계정 로그인 성공
- 주요 정적 화면 순회
- 범위 점검표의 각 항목을 검사하거나 제외 이유 기록
- 허용하지 않은 쓰기 요청 차단 확인
- `reports/report.html` 생성
